1. WHAT THIS PROJECT IS
========================================================================
Vinnoflow вЂ” a self-hosted "automation hub". v1.0 focuses on one module:
"TG Cloud" вЂ” unlimited-storage cloud file manager built on top of Telegram
(files are actually stored in a Telegram channel/chat; Telegram is used as
the storage backend). Has a web dashboard (React) + Node/Express backend +
MongoDB for metadata + a Telegram bot for uploading files via chat.
Future planned modules (per README): bot templates, Zapier-like automation
flows, AI agents вЂ” NOT built yet, only TG Cloud exists today.

========================================================================
2. TECH STACK
========================================================================
Backend:
  - Node.js (ESM, "type": "module"), Express 4
  - MongoDB + Mongoose (data: settings, folders, files, share links, bot config)
  - express-session (cookie-based auth sessions)
  - bcrypt (password hashing)
  - multer (multipart file upload handling, disk storage)
  - node-telegram-bot-api  в†’ the interactive Telegram BOT (commands, /settings, polling)
  - telegram (GramJS)      в†’ MTProto CLIENT (logs in as the bot via botAuthToken) used
                              for actual file upload/download/streaming to/from Telegram,
                              because Bot API alone can't handle large files/streaming well.
  - dotenv-style env handling via config/vars.js (env values also mirrored into MongoDB)

Frontend:
  - React 18 + Vite
  - react-router-dom (client-side routing)
  - Plain `fetch` / `XMLHttpRequest` for API calls (no axios), session-cookie auth
  - No CSS framework вЂ” plain index.css
  - Dev server on port 5173, proxies /api,/login,/logout,/setup,/profile,/cloud,
    /stream,/download,/share to backend on port 3000 (see frontend/vite.config.js)
  - In production, Express serves the built frontend/dist directly (see app.js)

Root:
  - npm workspaces-style but NOT a real pnpm/npm workspace вЂ” just a root
    package.json with scripts that shell out to backend/ and frontend/:
      postinstall в†’ installs deps in both backend/ and frontend/
      start       в†’ node backend/index.js            (production / single-process)
      dev         в†’ concurrently: vite dev server + node backend/index.js
      build       в†’ builds frontend (vite build в†’ frontend/dist)

4. BACKEND вЂ” FILE BY FILE (imports / exports / functions / routes)
========================================================================

--- backend/index.js ---
Entry point. Boot sequence (function `boot()`):
  1. Mounts settingsRoutes at /setup and /profile
  2. Connects MongoDB via config/db.js if MONGO_URI present
  3. refreshConfigFromDB() to sync DB-stored config into memory
  4. Loads tgwebcloud.js в†’ initRoutes() (registers all /cloud,/stream,/download,/share routes)
  5. If Telegram creds present: initClient() (MTProto) then reinitBot() (bot polling) +
     registerBotHandlers() (tgbotcloud.js command handlers)
  6. addCatchAll() (from app.js) вЂ” serves React SPA for all non-API paths
  7. app.listen(PORT)
IMPORTS: config/db.js, config/vars.js, app.js (app, addCatchAll), then dynamic
  imports of settingsRoutes.js, tgwebcloud.js, bot.js, tgbotcloud.js.

--- backend/app.js ---
Creates and configures the core Express `app`: JSON/urlencoded body parsers,
static serving of frontend/dist and backend/public, express-session config
(cookie: httpOnly, 7-day maxAge, secret from SESSION_SECRET env/config).
EXPORTS: `app` (default), `markSetupComplete()`, `checkSetupStatus()`,
  `requireAuth` (session-based middleware, unused directly вЂ” auth/auth.js is
  the one actually wired into routes), `addCatchAll()`.
ROUTES DEFINED HERE:
  GET  /api/me              в†’ current session user (displayName/email/avatarUrl)
  GET  /api/setup/status    в†’ { dbConnected, setupComplete }
  POST /login               в†’ checks Settings.admin (DB) OR EMAIL/PASSWORD env fallback
  GET  /logout              в†’ destroys session

--- backend/bot.js ---
Singleton wrapper around `node-telegram-bot-api`. Ensures old polling instance
is stopped before creating a new one (prevents duplicate polling / 409 conflicts
IF called correctly вЂ” see known issue note in В§6).
EXPORTS: default `botProxy` (Proxy that forwards method calls to the live
  bot instance, throws if not initialized yet), `reinitBot()`, `getBot()`.

--- backend/config/db.js ---
Mongoose connection helper.
EXPORTS: `connectDB(uri)` в†’ { ok, error? }, `isConnected()`.

--- backend/config/vars.js ---
Central config layer: merges process.env with values persisted in MongoDB
(so setup wizard writes credentials to DB + .env-equivalent storage).
EXPORTS: `getConfig()` (returns current merged config object: MONGO_URI,
  API_ID, API_HASH, BOT_TOKEN, TG_SESSION, EMAIL, PASSWORD, SESSION_SECRET,
  PORT, etc.), `refreshConfigFromDB()`, `writeEnvValues(obj)` (persists to
  .env-like store), `getBotToken()`.

--- backend/auth/auth.js ---
EXPORTS: default `authMiddleware(req,res,next)` вЂ” 401s unless
  `req.session.auth === true`. Used to protect /profile/update and other
  authenticated routes.

--- backend/features/settings/Models/SettingsModel.js ---
Mongoose model `Settings` (one singleton document holds all app config):
  setupComplete: Bool
  admin: { email, passwordHash, displayName, avatarUrl }
  telegram: { apiId, apiHash, botToken, dbChannelId, ownerId, session, connected }

--- backend/features/settings/settingsRoutes.js ---
Express Router mounted at both /setup and /profile.
Helper fns: getSettings() (find-or-create Settings doc), isSetupComplete(),
  autoInitTelegram() (calls tgwebcloud.initClient() + bot.js reinitBot() +
  tgbotcloud.registerBotHandlers() вЂ” this is what actually starts Telegram
  connections when config already exists in DB, i.e. auto-restore on reboot).
ROUTES:
  POST /setup/connect-db        в†’ save+connect MONGO_URI; if setup already
                                   complete in DB, auto-restores full session
                                   (marks setup complete, calls autoInitTelegram(),
                                   logs the user in)
  POST /setup/save-credentials  в†’ create admin email/password (blocked once setup done)
  POST /setup/save-profile      в†’ save displayName/avatar, marks setupComplete=true,
                                   calls autoInitTelegram(), logs user in
  POST /profile/update          [auth] в†’ update displayName/email
  POST /profile/change-password [auth] в†’ update password
  POST /profile/update-telegram [auth] в†’ save API_ID/API_HASH/BOT_TOKEN
  POST /profile/connect-telegram[auth] в†’ (re)initClient + reinitBot with new creds
  GET  /profile/settings-data   [auth] в†’ returns current settings for the Profile page

--- backend/features/tgcloud/Models/BotModels.js ---
  BOT: { folderId }                          вЂ” default upload folder for bot
  ActiveChannel: { channelId (unique), folderId } вЂ” maps a Telegram channel to a folder,
      used for multi-channel routing (channel posts land in matching folder)

--- backend/features/tgcloud/Models/FileModels.js ---
  Folder: { name, folderId (unique), parentId, channelId }
  FolderFile: { folderId, filename, fileId, messageId, channelId, uniqueId (unique),
                size, thumbId }
  ShareLink: { token (unique), uniqueId, fileId, filename, size }

--- backend/features/tgcloud/tgbotcloud.js ---
The interactive Telegram BOT's command/message handlers (built on the
`node-telegram-bot-api` instance from bot.js via `getBot()`).
IMPORTS: getBot (bot.js), Folder/FolderFile (FileModels), BOT/ActiveChannel
  (BotModels), getConfig (config/vars.js).
Helpers: fmtBytes(), sanitize(), extractFileInfo(msg) (pulls filename/fileId/
  size from document/video/audio/photo message types), file batching via
  `fileBatches` Map + BATCH_WAIT (3s) вЂ” groups multiple files sent together
  into one save operation.
KEY FEATURE: registerBotHandlers() вЂ” wires up bot commands: /start, /addf
  (register folder), /settings (inline-keyboard based settings menu вЂ” added
  in a previous round, replacing old text-command flow), plus generic
  message handler that intercepts document/video/audio/photo uploads sent
  to the bot or to a mapped channel and saves them into the matching Folder
  as FolderFile records. Multi-channel support: incoming channel posts are
  routed to whichever Folder is mapped via ActiveChannel.channelId.

--- backend/features/tgcloud/tgwebcloud.js ---
Owns the MTProto client (GramJS `TelegramClient`) AND all cloud HTTP routes.
IMPORTS: telegram (TelegramClient, Api), telegram/sessions (StringSession),
  telegram/client/uploads (CustomFile), multer, authMiddleware, Folder/
  FolderFile/ShareLink (FileModels), getConfig/refreshConfigFromDB/
  writeEnvValues (config/vars.js).
In-memory state: `senders` Map, `metaCache` Map, `chunkCache` (LRU-ish cache
  for streamed file chunks, 2min TTL, max 50 entries), `_client` (singleton
  MTProto client instance).
KEY FUNCTIONS:
  initClient()   в†’ creates/starts the GramJS TelegramClient using
                    botAuthToken (logs in AS the bot via MTProto, not a user
                    account), reuses/saves session string to DB+env so it
                    doesn't need to re-auth every boot. No-ops if already
                    connected. Skips if API_ID/API_HASH/BOT_TOKEN missing.
  reinitClient() в†’ disconnects + re-runs initClient() (used after credential
                    changes from Profile page)
  getClient()    в†’ returns current MTProto client instance
  initRoutes()   в†’ registers all Express routes below (multer disk-storage
                    upload middleware configured here вЂ” supports MULTIPLE
                    files per request, backward compatible with single-file)
ROUTES (mounted from initRoutes(), most require session auth):
  GET  /api/cloud/folders              в†’ list root-level folders
  GET  /api/cloud/folder-data/*        в†’ list a folder's subfolders + files
  POST /cloud/upload                   в†’ multi-file upload; streams progress
                                          via Server-Sent-Events-style chunked
                                          response (server-received % and
                                          Telegram-upload %) вЂ” uses `senders`
                                          map to track per-upload GramJS sender
  GET  /stream/:uniqueId               в†’ range-request video/audio streaming,
                                          reads chunks from Telegram via
                                          chunkCache/CHUNK(1MB)/PREFETCH(15)
  GET  /download/:filename             в†’ full file download from Telegram
  POST /cloud/share                    в†’ creates a ShareLink (token) for a file
  GET  /share/:token                   в†’ public (no-auth) share landing page
  GET  /share/stream/:token            в†’ public (no-auth) stream via share token

6. HOW EVERYTHING CONNECTS (data/feature flow)
========================================================================
- First run: no DB configured в†’ Setup.jsx wizard walks user through
  connect-db в†’ save-credentials в†’ save-profile. Last step marks setup
  complete in DB + calls autoInitTelegram() which starts BOTH Telegram
  connections (MTProto client for file I/O + bot polling for commands).
- Subsequent runs: index.js boot() reads config; if Telegram creds already
  known at boot time it starts them directly; if not yet loaded (e.g. env
  missing but DB has them), the frontend's initial `/setup/connect-db` call
  triggers the "existing setup detected вЂ” auto-restoring" path, which is
  what actually calls autoInitTelegram() in that scenario.
- File upload (web UI): CloudFolder.jsx в†’ POST /cloud/upload (multer saves
  to disk temporarily) в†’ tgwebcloud.js reads temp file в†’ GramJS `_client`
  uploads it into Telegram (bot's own chat/channel) в†’ FolderFile record
  saved in MongoDB (fileId/messageId/uniqueId) в†’ temp file deleted.
- File upload (via Telegram bot directly): user sends a file to the bot (or
  posts in a mapped channel) в†’ tgbotcloud.js message handler в†’ extractFileInfo()
  в†’ looks up target Folder via BOT.folderId or ActiveChannel mapping в†’
  creates FolderFile record. NOTE: bot.js uses Bot API for receiving/sending
  messages; tgwebcloud.js's GramJS client is used for the heavier lifting
  (large file streaming/download) вЂ” both point at the SAME underlying bot
  account (one uses Bot API token directly, the other logs the bot in via
  MTProto using that same token as botAuthToken).
- Streaming/download/share: tgwebcloud.js routes fetch the file from
  Telegram via the GramJS client (chunked, cached in chunkCache) and pipe it
  to the browser вЂ” this is why large files/video seeking work despite
  Telegram Bot API's normal file-size limits.
- Auth: express-session cookie set on /login or during Setup's final step;
  authMiddleware (auth/auth.js) protects /profile/* and most /cloud/*,
  /stream, /download routes; ProtectedRoute (frontend) blocks page access
  client-side, backed by real server-side 401s.

7. KNOWN OPERATIONAL NOTES (useful context, not code bugs)
========================================================================
- Telegram bot polling uses node-telegram-bot-api; MTProto file I/O uses
  gramjs (`telegram` package) вЂ” this is intentional, NOT duplicate libraries
  for the same job. Do not "simplify" to one library without understanding
  both are required for different capabilities (bot commands vs large-file
  MTProto access).
- A `409 Conflict: terminated by other getUpdates request` error means the
  SAME bot token is being polled from two places at once (e.g. this app
  running in two environments simultaneously) вЂ” not a code bug by itself,
  but be careful not to introduce a code path that calls reinitBot()/
  initClient() twice concurrently (bot.js's reinitBot() does stop the old
  poller first, but only if called sequentially вЂ” concurrent calls could race).
- Repeated GramJS "Not connected"/ECONNABORTED reconnect churn is commonly
  caused by the host device/OS suspending background network (e.g. mobile
  device battery optimization killing a background terminal's sockets), not
  a bug in tgwebcloud.js.
